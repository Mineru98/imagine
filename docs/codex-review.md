# Codex Review — imagine 하네스

## 전체 아키텍처 평가
이 저장소는 Claude Code 플러그인 위에 다수의 스킬과 에이전트를 얹고, 실행 공통부를 `oauth-session.js`와 재시도 정책으로 묶는 구조다. 강점은 역할 분리가 선명하다는 점이다. 기본 `imagine`은 범용 생성/편집, `image-to-code`는 마크업 변환, `imagine-hero`와 `imagine-thumb`는 산출물 규격을 강하게 고정한다. 출력 경로를 프로젝트 내부로 한정하고 설치 전제도 명시해 운영 경계도 분명하다. 다만 OAuth 계층은 “실제 브리지”보다 포트 점유와 세션 파일 상태 판단에 더 가까워 보여, README의 약속과 런타임 실체 사이 간극이 남는다.

## 스킬 동작 방식의 일관성
스킬 설계 원칙은 대체로 일관적이다. `imagine`은 enum 기반 파라미터와 출력 경로를 고정하고, `image-to-code`는 애매한 요청에서 1턴만 질문한 뒤 분기한다. `imagine-hero`는 “이미지 내부 텍스트 금지”와 세이프존 40%를 계약으로 선언하고, `imagine-thumb`는 제목 텍스트를 모델 프롬프트에 넣지 않고 후처리로 합성한다. 각 스킬이 “무엇을 자동화하고 무엇은 금지하는가”를 먼저 잠근 점은 좋다. 반면 negative prompt, 출력 규약, 재생성 상한이 여러 파일에 반복돼 drift 위험이 있다. `image-to-code`만 설계 문서 § 참조를 쓰고 다른 스킬은 자체 규약 중심이라 기준 문서의 단일성도 약하다.

## 에이전트 책임 모델 평가
`prompt-director`와 `visual-critic`의 책임 분리는 좋다. 전자는 “한 번만 번역·보정”과 “스타일 주입 금지”를 두고, 후자는 자동 실행과 자동 재생성을 모두 금지한다. 이 조합은 생성 전 보정과 생성 후 평가의 충돌을 막는다.

`agents/visual-critic.md:12`
```md
- **자동 실행되지 않는다.**
- **사용자 명시 호출만**.
- **Prompt Director와 동시 자동 실행 금지.**
```

다만 이 책임 모델은 문서 수준에서 강하고, 실행 코드에서 실제 강제되는지는 확인하지 못했다.

## OAuth 프록시 의존의 리스크

## 개선 제안 Top 5
1. OAuth 프록시 계약을 문서가 아니라 코드 인터페이스로 승격해야 한다. 실제 upstream 연결 확인용 헬스체크와 실패 원인 코드를 분리하라.
2. 스킬 공통 정책을 단일 규약 파일로 묶어라. negative prompt, retry 상한, 출력 경로 규칙이 각 SKILL.md에 중복돼 유지비가 커진다.
3. 에이전트 금기 사항은 실행 스크립트에서 검증해야 한다. 현재는 책임 경계가 문서 중심이라 회귀에 취약하다.
4. `image-to-code`와 imagine 계열의 라우팅 기준을 테스트 가능한 decision table로 내리면 트리거 충돌을 줄일 수 있다.
5. README의 “무제한에 가깝게” 같은 표현은 운영 리스크를 축소해 보일 수 있으므로, 쿼터·세션 만료·포트 충돌을 초기 소개에도 같이 드러내는 편이 낫다.

## 위험 신호 (Red Flags)
- README의 가치 제안이 강한 반면, 리뷰 범위 구현에서 실제 OAuth 브리지 동작은 확인되지 않았다.
- 스킬/에이전트 계약이 풍부하지만 대부분 문서 규범이라, 코드 강제가 약하면 쉽게 무너질 수 있다.
- `image-to-code`는 설계 문서 의존이 보이는데, imagine 계열은 개별 문서 규약 중심이라 장기적으로 기준 불일치가 생길 수 있다.
- 마케팅 문구가 운영 제약보다 앞서 있다. 인증 만료, 포트 충돌, rate limit는 부가 예외가 아니라 핵심 실패 축이다.
- 확인하지 않음: 실제 `generate.js`, `edit.js`, `run.js`, 훅/오케스트레이터 구현, 설계 문서 본문.
